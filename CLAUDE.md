# CLAUDE.md

## Project at a glance

This is **scatterpub-toolchain-example** — a worked example project for the [scatterpub-toolchain](https://github.com/scattercode/scatterpub-toolchain).

It contains a small sample of real scanned pages from *Tell Me, Bella* by Vahan Totovents (published by the Armenian Institute), ready to run through the full digitisation pipeline: OCR → clean → copy-edit → Word export.

Follow `TUTORIAL.md` for a step-by-step walkthrough.

The toolchain (scripts, Claude Code skills, Python dependencies) lives in the `toolchain/` git submodule. See `toolchain/CLAUDE.md` for full script and skill documentation.

## Language

All content, comments, and documentation should use **British English** (e.g. 'colour' not 'color', 'organise' not 'organize', 'travelling' not 'traveling'). Use the Oxford comma in lists.

## Repository structure

```
toolchain/                       git submodule — scatterpub-toolchain (scripts + skills)

publishing/tell-me-bella/
  book.md                        Book metadata (title, author)
  ocr/scans/clean/               Sample clean scan PDFs — tracked in git
  ocr/<slug>-raw.md              Assembled OCR output (generated by scripts)
  ocr/<slug>-clean.md            Cleaned OCR output (generated by scripts)
  ocr/<slug>-ai-clean.md         AI OCR-corrected version (generated by /copyeditor)
  draft/<slug>.md                Working copy for manual editing — tracked in git
  draft/<slug>.docx              Word document for Vellum import (generated by scripts)
  review/<slug>-review.html      Copy-edit review (generated by /copyeditor)
```

## Workflow

The pipeline separates generated files from manual work:

1. **`ocr/`** — machine output. Files here are always safe to overwrite; they can be regenerated by re-running the scripts.
2. **`draft/`** — human work. After the AI OCR pass, the user copies `ocr/<slug>-ai-clean.md` into `draft/` and performs manual editing there. Files in `draft/` are tracked in git and represent work in progress.

**Guardrail:** Before writing any file into a `publishing/*/draft/` folder, check whether a file with that name already exists. If it does, stop and tell the user — do not overwrite it silently. The user must explicitly confirm before you proceed. This does not apply to files in `ocr/`, `review/`, or any other folder.

## Setup

```bash
git clone <this-repo>
git submodule update --init
brew install poppler tesseract pandoc
cd toolchain && poetry install
```

## Running scripts

Run from the repository root:

```bash
# OCR the clean scans → raw Markdown
toolchain/.venv/bin/python toolchain/scripts/ocr-to-markdown.py \
  "publishing/tell-me-bella/ocr/scans/clean"

# Clean the raw output
python3 toolchain/scripts/clean-ocr.py \
  "publishing/tell-me-bella/ocr/tell-me-bella-raw.md" --join-hyphens

# Generate Word document for sharing or Vellum import
python3 toolchain/scripts/md-to-docx.py \
  "publishing/tell-me-bella/draft/tell-me-bella.md"
```

## Skills

Skills are provided via symlinks in `.claude/skills/` pointing into the `toolchain/` submodule. They are available automatically once the submodule is initialised — no separate workspace setup is needed.

```
/copyeditor      — copy-edit manuscripts; style guide determined by language in book.md
/pdf             — general-purpose PDF processing
/skill-creator   — create and improve skills
```

## What not to edit

- `publishing/*/ocr/*.md` — generated by the pipeline scripts; re-run the scripts to regenerate.
- `publishing/*/review/` — generated outputs; re-run the relevant script or skill to regenerate.
- `publishing/*/draft/*.md` — **do not overwrite without confirming with the user** (see Guardrail above).
- `publishing/*/draft/*.docx` — generated from the draft Markdown; safe to overwrite.

`publishing/*/book.md` **should** be edited — it is the source of truth for book title and author metadata.


## Git hooks

Git hooks are managed by [Lefthook](https://lefthook.dev) (`lefthook.yml`).

- **commit-msg** — enforces [Conventional Commits](https://www.conventionalcommits.org/)
  via the shared script at `.githooks/commit-msg`.

There is no pre-commit linting, as this repo is mostly manuscript prose.

Conventional Commits format: `<type>[(scope)][!]: <description>`

- Types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`,
  `ci`, `chore`, `revert`.
- The description is non-empty with no trailing full stop; the header is 100
  characters or fewer (72 or fewer preferred).
- A body, if present, is separated from the header by a blank line.

After cloning, install the hooks once with `lefthook install` (requires the
`lefthook` binary, e.g. `brew install lefthook`).
